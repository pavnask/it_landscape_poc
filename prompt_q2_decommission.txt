- архитектурно нужен controlled decommission, а не просто выключение компонента

Ниже reference patterns:
1. Decommission starts from dependencies
Pattern: Вывод из эксплуатации начинается не с удаления UI, а с анализа зависимостей, миграции потребителей и отключения интеграций.
Example: Сначала карта зависимостей и миграция вызовов, затем отключение сервиса и только потом удаление компонента.

2. Component inside platform
Pattern: Если компонент входит в более крупную платформу, decommission должен учитывать связи: платформа → подсистема → функция → сервис → интеграции.
Example: Нельзя выключать подсистему внутри платформы без проверки всей цепочки зависимостей.

3. Parallel migration then shutdown
Pattern: При замене сервиса или интеграции нужно сначала ввести новый контур параллельно старому, перевести потребителей и только потом отключать старый.
Example: Сначала parallel run, затем migration и validation, потом final shutdown.
