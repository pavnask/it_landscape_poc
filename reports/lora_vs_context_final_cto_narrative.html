1. Контекст и гипотеза

Исходная идея

Проверялась гипотеза: можно ли через LoRA обучить модель быть «архитектором компании», который стабильно применяет наши паттерны интеграции и reasoning-подходы.

Если обучить модель через LoRA на наших архитектурных паттернах, она сможет работать как архитектор без RAG.
  • Kafka, integration layer, API gateway
  • Архитектурные trade-offs
  • Decommission и dependency reasoning

2. Что было сделано

Данные и подход

  • Реальный enterprise-ландшафт из YAML/JSON
  • Reasoning datasets v1, v2, v3 PRO
  • Messages-format training для свободных ответов архитектора
  • RAG на реальных фрагментах ландшафта

Кейсы

  • AI assistant
  • ERP ↔ Mobile integration
  • Decommission news editor

4. Ключевой результат

D ≈ F. Когда модели дают тот же самый контекст через RAG, LoRA не даёт дополнительного прироста качества.

Перевод на язык принятия решений: значительная часть эффекта, который ранее казался эффектом LoRA, на самом деле объясняется качественным retrieval-контекстом.

Вариант Что показывает Наблюдение
D Контекст + паттерны без LoRA Даёт корректное решение, но без глубины reasoning
F Тот же контекст + LoRA Почти тот же уровень качества; LoRA не обгоняет retrieval

5. Что мы узнали про LoRA

  • LoRA улучшает стиль ответа
  • LoRA повышает консистентность формулировок
  • LoRA может подталкивать модель к корпоративным паттернам
  • Но LoRA не становится источником знаний о системе
LoRA не учит думать глубже. Она учит говорить правильным тоном.

6. Что мы узнали про RAG

  • RAG приносит в inference реальные знания о ландшафте
  • RAG даёт зависимость от актуальных систем, сервисов и ограничений
  • RAG объясняет существенную часть прироста качества
Знания находятся в контексте. Не в весах.

7. Настоящее ограничение

Эксперимент показал, что основное ограничение находится не в LoRA и не в retrieval. Ограничение — в reasoning-capability самой базовой модели.

Проблема Что видно в ответах Вывод
Поверхностное reasoning Модель читает паттерны, но применяет их неглубоко Нужна более сильная базовая модель
Слабый dependency reasoning Decommission остаётся слабым даже с dataset v3 PRO 1.5B модель упирается в capability ceiling
Trade-offs без строгости Есть стиль, но нет убедительной глубины решения Fine-tuning не заменяет reasoning

8. Рекомендация

Что не делать

  • Не использовать LoRA как основной механизм «обучения архитектора»
  • Не пытаться «зашить весь ландшафт в веса»

Что делать

  • RAG как knowledge layer
  • Более сильная базовая модель как reasoning layer
  • LoRA — как optional enhancement для стиля и консистентности

10. Сильные утверждения

  • LoRA не является источником знаний.
  • Качество архитектурного решения определяется контекстом, а не весами модели.
  • Fine-tuning не заменяет reasoning.
  • Мы не упёрлись в технологию. Мы упёрлись в границу текущей модели.
  • Правильная ставка: RAG + более сильная модель. LoRA — только как усилитель стиля.
Это не провал POC. Это точная локализация того, где именно находится ценность — и где её нет.
