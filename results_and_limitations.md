# Results & Limitations (POC: LoRA для генерации архитектурных паттернов)

## 1. Базовые понятия

### Что такое LoRA
LoRA (Low-Rank Adaptation) — метод дообучения LLM:
- базовая модель не меняется
- обучаются только адаптеры (~1%)
- дешево и быстро

В нашем POC:
- Qwen 1.5B: ~40 минут
- Mistral 7B: ~21 час

---

### LoRA vs RAG

| Подход | Роль |
|--------|------|
| LoRA | обучает паттернам |
| RAG | дает реальные данные |

---

## 2. Гипотеза

Малую модель можно научить:
- понимать IT-ландшафт
- генерировать архитектуру
- соблюдать JSON-контракт

---

## 3. Результаты

### Как читать метрики (ВАЖНО)

Формат:
- **X / N = количество полностью успешных кейсов**
- кейс считается успешным, если:
  - JSON валиден
  - есть `elements` и `relations`
  - типы связей корректны
  - нет лишних wrapper-структур

Примеры:
- **5/12** → 5 корректных архитектур из 12 тестов
- **0/12** → ни одного полностью валидного результата
- **3/10** → 3 корректных кейса из 10 реальных сценариев

👉 важно: даже при 0/12 модель может "понимать задачу", но не проходит строгую валидацию

---

### Synthetic benchmark (искусственные сценарии)

| Модель | Результат |
|--------|----------|
| baseline | 3/12 |
| LoRA (Qwen, v1) | **5/12** |
| LoRA (SEAF structured) | **0/12** |

---

### Real benchmark (реальные enterprise сценарии)

| Модель | Данные | Результат |
|--------|--------|----------|
| LoRA (synthetic only) | synthetic | 2/10 |
| LoRA (+real YAML) | real | 0/10 |
| LoRA (+SEAF structured) | structured real | **3/10** |

---

### Интерпретация

- **5/12 (synthetic)** → модель учится паттернам  
- **0/12 (SEAF → synthetic test)** → потеря alignment с synthetic  
- **3/10 (real)** → улучшение на реальных задачах  

👉 ключевой вывод:  
**качество real-данных важнее synthetic accuracy**

---

## 4. Ключевой инсайт

Модель:

- ✅ понимает архитектурные паттерны  
- ✅ переносит знания на реальные сценарии (3/10)  
- ❌ не соблюдает строгий JSON-контракт  
- ❌ генерирует некорректные типы связей  

👉 основной bottleneck — **не интеллект модели, а контроль вывода**

## 6. Как будет работать правильная архитектура

### Шаг 1 — RAG (контекст)

```json
{
  "elements": [
    {"id": "app_checkout"},
    {"id": "cap_payments"}
  ]
}
```

---

### Шаг 2 — LoRA (генерация)

```json
{
  "elements": [
    {"id": "api_payment", "type": "api"}
  ],
  "relations": [
    {"source": "app_checkout", "type": "uses", "target": "api_payment"}
  ]
}
```

---

### Шаг 3 — Structured Output (контроль)

Исправляет:
- JSON
- missing fields
- типы связей

## 7. Ограничения

- LoRA ≠ строгий формат
- Synthetic ≠ Real
- малые модели нестабильны

---

## 8. GO / NO-GO (рекомендация CTO)

### NO-GO

❌ Использовать только LoRA  
Причины:
- нестабильный JSON
- ошибки в типах связей
- невозможность автоматизации

---

### GO (рекомендуемая архитектура)

✅ LoRA + RAG + Structured Output

Почему:

1. LoRA уже доказал:
   - обучение паттернам (5/12)
   - перенос на real (3/10)

2. Основные ошибки:
   - формат
   - валидация
   - типы связей

👉 это НЕ проблема модели, а проблема output-контроля

---

### Ожидаемый эффект следующего этапа

Без retraining:

- текущие **3/10 → 5–6/10**
- за счёт:
  - JSON repair
  - schema validation
  - relation rules

## 11. Финальный экспериментальный итог

### Что улучшилось

- переход от synthetic → real → structured (SEAF) данных
- рост качества на реальных сценариях:  
  **2/10 → 3/10**

### Что ухудшилось

- падение synthetic benchmark:  
  **5/12 → 0/12**

### Почему это нормально

- модель сместилась ближе к реальному распределению данных
- synthetic benchmark перестал быть репрезентативным

### Главный вывод

👉 модель начинает работать на реальных задачах,  
но требует обязательного слоя валидации
